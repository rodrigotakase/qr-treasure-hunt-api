# qr-treasure-hunt-api

PHP/MySQL backend for a QR code treasure hunt. Treasures are hidden in the
real world, each labelled with a QR code. Players scan them with the
[ar-treasure-hunt-client](../ar-treasure-hunt-client) app, which calls this
API to identify treasures, register collections and show leaderboards.

No framework: a front controller, a small router and a few classes on top of
PDO. The only Composer dependency is
[endroid/qr-code](https://github.com/endroid/qr-code). Built to run on cheap
shared hosting (Hostinger), where deploying means copying files.

Requires PHP 8.1+ with `pdo_mysql` and `gd`, MySQL/MariaDB and `mod_rewrite`.

## Live demo

- Admin panel: https://treasurehunt.kaiguritech.com/admin
- API: https://treasurehunt.kaiguritech.com/api/hunts/1/treasures
- Leaderboard: https://treasurehunt.kaiguritech.com/api/hunts/1/leaderboard

## Setup

1. `composer install`
2. `cp .env.example .env` and fill in the database credentials.
   `CLIENT_BASE_URL` is the client app URL: it is what the QR codes contain,
   and its origin is allowed for CORS.
3. Create the tables by running `migrations/001_init.sql` in phpMyAdmin
   (there is no migration runner, it's a single SQL file).

Only the contents of `public_html/` (front controller, `.htaccess`, CSS) go
in the web root. Everything else (`src/`, `vendor/`, `.env`) must stay
outside it, normally one level above. If the web root has no private parent,
which is the case for Hostinger subdomains, put the app files in any private
folder and point `$appDir` at the top of `index.php` to it.

## API

All responses are JSON. Errors come back as `{ "error": "..." }`.

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/treasures/{id}` | Treasure info. With `?user_id=`, includes `location` if that user already collected it. |
| GET | `/api/hunts/{id}/treasures` | Treasures of a hunt with `collected` state for `?user_id=`. Uncollected ones omit `id` and `location`. |
| POST | `/api/collect` | Body: `{ "user_id", "nickname", "treasure_id" }`. Upserts the user and registers the collection. Scanning twice returns `alreadyCollected: true`, not an error. |
| GET | `/api/hunts/{id}/leaderboard` | Players ranked by treasures collected, earlier finisher wins ties. |

User ids are UUIDs generated by the client device (no signup). Treasure ids
are UUIDs generated by the server when a treasure is created, and are what
the QR codes carry.

## Admin panel

`/admin` manages hunts and treasures, shows the leaderboards and generates
each treasure's QR code as a downloadable PNG.

There is no login. The pages send `noindex`, but anyone with the URL can get
in, so protect `/admin` with HTTP Basic Auth before using it for real.
